## Background

The v0.1.0 observability target calls for practical throughput visibility in `/api/metrics`. NetSentry already exports packet and alert counters, current/high-water queue depth, rule match latency histograms, alert write latency histograms, storage gauges, worker counters, and capture heartbeat gauges. What is still missing from the core local Prometheus view is a simple actual processing-rate signal derived from engine counters.

## Goal

Expose process uptime plus packet and alert per-second gauges from `/api/metrics`, computed from existing process-local counters, and document the metric scope without claiming production PPS capacity.

## Scope

- Add a `StartedAt` timestamp to stats snapshots.
- Export `netsentry_process_uptime_seconds`.
- Export `netsentry_packets_processed_per_second`.
- Export `netsentry_alerts_generated_per_second`.
- Add focused tests for the new gauges and HELP text.
- Update public documentation to describe the new current rate gauges.

## Non-Goals

- No worker-utilization model.
- No sliding-window rate implementation.
- No Prometheus client dependency.
- No rule, storage, receiver protocol, C capture, Docker, release tag, or registry publishing change.
- No performance claim beyond local counter-derived gauges.

## Validation Plan

## Risk Points

- Very early scrapes can produce near-zero uptime; code must avoid divide-by-zero.
- Rate gauges are process-lifetime averages, not sliding-window throughput.
- Documentation must not present these gauges as production throughput guarantees.

## Rollback Plan

Revert the stats/API metric additions, tests, and docs if the gauges produce unstable output or regress existing metrics.
