# Task Plan: R90-06 release decision package

## Metadata

- Timestamp: 2026-07-16T16:45:08Z
- Branch: main
- Risk Level: High

## Goal

Assemble a reconciled v0.1.1 release decision package containing the intended
version, candidate commit, reviewed evidence, local artifact checksum, workflow
behavior, and an explicit publication decision.

## Scope

- Make tag workflows select the release evidence record matching the tag
  version instead of always using v0.1.0.
- Make packaged release notes describe their actual version.
- Validate and push one candidate implementation commit.
- Build v0.1.1 artifacts from that exact candidate and record their checksum.
- Publish a sanitized decision package with publication held pending separate
  authorization.
- Reconcile the roadmap and task state.

## Non-Goals

- Do not create or move a tag.
- Do not trigger GitHub Release or GHCR publication.
- Do not grant final publication authorization.
- Do not include local artifact bytes, private paths, credentials, or operator
  details in Git or the Vault.

## Risks

- A version tag can validate the wrong evidence record.
- A generated archive can describe a stale hard-coded release version.
- A checksum can be recorded against a commit other than the proposed tag
  target.
- Release-gate acceptance can be mistaken for publication authorization.

## Validation

- Workflow syntax and pinned supply-chain checks.
- `VERSION=0.1.1 make rc-check`, including Docker.
- `RELEASE_EVIDENCE=docs/evidence/release-v0.1.1.md make release-gate`.
- Artifact checksum verification for the exact candidate commit.
- `make docs-check`, `make python-check`, `make knowledge-check`, JSON parsing,
  `git diff --check`, and sensitive-information review.

## Acceptance Criteria

- Tag workflows derive the evidence record from the release tag/version.
- v0.1.1 artifact name and checksum are recorded against one verified
  `origin/main` candidate commit.
- The decision package links the reviewed evidence and states `hold` for tag
  creation/publication.
- The increment is pushed, fetch-verified, and synchronized to the Vault.

## Stop Conditions

Stop before tag or publication actions, if the candidate commit/checksum cannot
be reconciled, or if any validation result is ambiguous.
