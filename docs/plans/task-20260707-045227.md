## Background

The roadmap still lists the full graceful shutdown sequence as a remaining W10 validation gap. The receiver already supports `Stop` and `Wait`, and the pipeline worker exits on context cancellation, but `cmd/netsentry` currently returns as soon as the shutdown context is cancelled. That can close the SQLite store through `defer` before the worker goroutine has finished unwinding.

## Goal

Make the engine entrypoint wait for receiver shutdown and pipeline worker exit after SIGINT/SIGTERM cancellation, and verify the shutdown orchestration with targeted Go tests plus the existing Makefile quality gates.

## Scope

- Refactor the engine entrypoint just enough to make shutdown ordering explicit.
- Start the pipeline worker with a waitable goroutine.
- On shutdown, stop the UDS receiver, wait for receiver goroutines, then wait for the worker before returning from `main`.
- Add a focused unit test around the shutdown orchestration if the current main package can be tested without running a full process.

## Non-Goals

- No protocol, rule, storage, API, or Docker behavior changes.
- No multi-worker pipeline implementation.
- No restart-free storage recovery or automatic disk cleanup.
- No release tag or registry publication.

## Validation Plan

Targeted tests should prove the waitable worker helper returns after context cancellation. `make test` should cover the broader C/Go regression suite. Documentation checks and dependency checks should remain green.

## Risk Points

- Incorrect ordering could hang shutdown if a goroutine does not observe context cancellation.
- Waiting in `main` must not close the SQLite store before the worker exits.
- Tests should avoid depending on real OS signals where a direct context cancellation is enough.

## Rollback Plan

Revert the `cmd/netsentry` shutdown helper and doc edits if shutdown tests or existing regression tests expose a hang or behavior regression.
