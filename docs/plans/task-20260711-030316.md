## Background

v0.1.0 has passed local synthetic fuzz and repeat-pcap checks but lacks a standardized repository-provided sanitized corpus. The current `scripts/gen_test_pcap.py` creates only the quickstart six-packet pcap. The release plan also requires external sustained fuzz evidence and realistic corpus pressure/query evidence.

## Goal

Add a deterministic, stdlib-only generator that writes a documented set of sanitized Ethernet pcap samples, expose it through the native Makefile, and prepare—but do not yet approve—the temporary gate exception that allows C fuzz evidence collection while deferring traffic-pressure evidence.

## Scope

- Add `scripts/gen_sanitized_corpus.py`.
- Add `make gen-sanitized-corpus`.
- Generate deterministic private-address, documentation-domain, non-secret payloads under a caller-selected directory.
- Add tests/documentation for generation, corpus handling, approval record, fuzz evidence, and release sequence.
- Keep generated binary pcap files outside Git by default.

## Non-Goals

- No C parser, UDS protocol, SQLite schema, API response, or runtime behavior changes.
- No automatic gate relaxation without administrator approval.
- No claim that synthetic fuzz closes external fuzz evidence.
- No `v0.1.0` tag or formal release in this task.
- No third-party packet library or new dependency.

## Execution Steps

1. Add the deterministic corpus generator and Makefile target.
2. Generate a temporary corpus outside the repository and validate its files/metadata.
3. Update public docs and knowledge notes with the path contract and approval boundary.
4. Request/record administrator approval for temporary fuzz-only gate exception.
5. After approval, run sustained C fuzz and record evidence; leave traffic pressure gate deferred.
6. Run applicable regression gates, review secrets and generated artifacts.
7. Create one atomic commit and push over the verified SSH remote.

## Validation Plan

The generator is accepted when repeated runs produce identical hashes, every file is parseable by the existing capture path, no output contains real external paths/credentials, and `make e2e-smoke` remains green. Release acceptance is not complete until the approval record and fuzz evidence exist.

## Risk Points

- A synthetic corpus can create false confidence; it must not be recorded as external fuzz evidence.
- Payloads must not contain real credentials, public target addresses, or operator paths.
- Temporary gate relaxation must be explicit, time-bounded, and limited to fuzz evidence.
- Generated binaries must remain outside Git and be reproducible from the script.

## Rollback Plan

Delete only the new generator/Makefile/docs changes and generated temporary corpus; restore the prior release gate wording. Do not alter parser or engine code. If validation fails three times, record a build failure log and stop per governance rules.

## Acceptance Criteria

- `make gen-sanitized-corpus CORPUS_DIR=/tmp/netsentry-sanitized-corpus` succeeds.
- At least three deterministic pcap samples and three matching pcapng samples are generated with a manifest.
- Repeated generation yields the same SHA-256 hashes.
- `make python-check`, `make e2e-smoke`, `make docs-check`, `make shell-check`, `make deps-check`, `make test-coverage`, and `make lint` pass.
- No gate is marked relaxed until administrator approval is recorded.
